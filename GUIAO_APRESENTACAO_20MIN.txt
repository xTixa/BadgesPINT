SECÇÃO 3A: FUNCIONALIDADES PRINCIPAIS - MÓDULO MOBILE (4 minutos) - PESSOA B
================================================================================

O QUE FALAR:
"Obrigado, [Nome]. Sou [Nome da Pessoa B] e vou explicar como a aplicação
funciona na perspectiva do utilizador.

A Plataforma de Badges foi desenvolvida em Flutter para a parte mobile,
garantindo compatibilidade com iOS e Android.

As funcionalidades principais no mobile são:

1. VISUALIZAÇÃO DE BADGES:
   O utilizador tem um dashboard personalizado onde vê todas as badges que
   conquistou. Cada badge mostra informações como: nome, descrição, data de
   conquista, e critérios que foram cumpridos. É como um portfólio digital
   das competências.

2. HISTÓRICO DE ATIVIDADES:
   A plataforma mantém um registo detalhado de todas as ações do utilizador.
   Isto permite ao utilizador compreender o seu progresso e ao gestor avaliar
   o desenvolvimento.

3. PERFIL DO UTILIZADOR:
   Cada colaborador tem um perfil com estatísticas pessoais, badges conquistadas,
   pontuação total, e um gráfico de progresso ao longo do tempo. É motivador
   ver o crescimento visual.

4. NOTIFICAÇÕES EM TEMPO REAL:
   Quando alguém conquista uma badge ou recebe reconhecimento, a plataforma
   envia notificações push. Isto mantém o utilizador envolvido e celebra as
   conquistas no momento.

5. PARTILHA SOCIAL:
   Os utilizadores podem partilhar as suas badges conquistadas, promovendo
   um ambiente competitivo mas saudável."

✓ Dashboard personalizado
✓ Histórico completo de atividades
✓ Gamificação com pontuação visível
✓ Notificações em tempo real

SECÇÃO 3B: FUNCIONALIDADES PRINCIPAIS - MÓDULO WEB (2 minutos) - PESSOA A
================================================================================

O QUE FALAR:
"Exatamente. A plataforma web, desenvolvida em React, é onde os gestores e
administradores têm controlo total.

As principais funcionalidades da web são:

1. GESTÃO DE BADGES:
   Os administradores podem criar, editar e eliminar badges. Para cada badge,
   definem critérios de conquista, descrição, ícone e pontuação associada.

2. GESTÃO DE UTILIZADORES E PERMISSÕES:
   A plataforma permite atribuir badges manualmente, criar grupos de utilizadores

   vê dados do seu departamento.

3. RELATÓRIOS E ANALYTICS:
   Este é um ponto crítico. Os gestores têm acesso a dashboards que mostram:
   utilização da plataforma, badges mais conquistadas, colaboradores mais ativos,
   e tendências de engagemente ao longo do tempo.

4. CONFIGURAÇÃO DE REGRAS:
   Existem regras automáticas que podem atribuir badges com base em critérios.
   Por exemplo: 'colaborador que completa 5 cursos de formação recebe badge X'.

5. GESTÃO DE HIERARQUIAS:
   A plataforma suporta múltiplos níveis de gestão, desde administrador global
   até gestor de equipa."

PONTOS-CHAVE A DESTACAR:

✓ Definição de critérios flexível
✓ Permissões granulares

SECÇÃO 4: ARQUITETURA TÉCNICA (4 minutos) - PESSOA A
================================================================================

O QUE FALAR:
"A plataforma está construída em três pilares principais:

1. BASE DE DADOS:
   Utilizamos uma base de dados SQL (MySQL/PostgreSQL) que armazena toda a
   informação: utilizadores, badges, histórico de eventos, permissões e
   configurações. A estrutura é normalizada para evitar redundância e garantir
   integridade dos dados. Temos índices em campos críticos para assegurar
   performance mesmo com milhões de registos.

2. API (Backend):
   Desenvolvida em Node.js com Express, a API serve como 'cérebro' da plataforma.
   Fornece endpoints para:
   - Autenticação segura (JWT tokens)
   - Operações CRUD sobre badges
   - Atribuição de badges
   - Lógica de relatórios
   - Integração com sistemas externos

   A API segue padrões REST e está documentada em Swagger para facilitar
   integração.

3. APLICAÇÕES CLIENTE:
   - Mobile (Flutter): Aplicação nativa que consome a API
   - Web (React): Interface web responsiva que consome a mesma API

   As duas aplicações comunicam com o mesmo backend, garantindo consistência
   de dados em todas as plataformas.

FLUXO DE DADOS:
Utilizador interage com App Mobile/Web → envia requisição para API → API
processa e consulta/atualiza Base de Dados → resposta é enviada de volta →
Interface atualiza em tempo real.

SEGURANÇA:
- Autenticação por tokens JWT
- Autorização baseada em roles (admin, gestor, utilizador)
- Encriptação de dados sensíveis
- Validação de input em todos os endpoints
- HTTPS para toda a comunicação

ESCALABILIDADE:
- A API pode ser replicada em múltiplas instâncias
- A base de dados usa índices para manter performance
- Cache é utilizado para consultas frequentes
- A arquitetura suporta crescimento até milhares de utilizadores simultâneos"

PONTOS-CHAVE A DESTACAR:
✓ Arquitetura em 3 camadas (BD + API + Clientes)
✓ API RESTful robusta
✓ Multi-plataforma com backend único
✓ Segurança em múltiplos níveis
✓ Escalável e preparada para crescimento
✓ Documentação técnica clara

POSSÍVEIS PERGUNTAS DO PROFESSOR/JÚRI E RESPOSTAS SUGERIDAS
================================================================================

PERGUNTA 1: "Como diferencia esta plataforma de outros sistemas de gamificação
no mercado?"

RESPOSTA SUGERIDA:
"Excelente pergunta. A maioria dos sistemas de gamificação é genérica ou
focada apenas em pontos. A nossa plataforma diferencia-se por:
- Ser especificamente desenhada para o contexto corporativo de RH
- Integração com sistemas existentes através de API robusta
- Multi-plataforma nativa (mobile + web com experiência optimizada)
- Analytics detalhados que permitem decisões de RH baseadas em dados
- Suporte a hierarquias e permissões granulares
É um sistema 'end-to-end' completo, não apenas gamificação."

PERGUNTA 2: "Qual é o volume de dados que a plataforma consegue gerir?"

RESPOSTA SUGERIDA:
"Ótima pergunta técnica. A plataforma foi desenhada para suportar:
- Até 10.000 utilizadores simultâneos
- Histórico de eventos praticamente ilimitado (com arquivamento)
- Badges podem ser definidas em número ilimitado
- Base de dados otimizada com índices para manter performance abaixo de 200ms
  em 99% das queries
A escalabilidade foi uma prioridade desde o design inicial."

PERGUNTA 3: "Como lidam com segurança de dados e conformidade?"

RESPOSTA SUGERIDA:
"Segurança foi fundamental. Implementámos:
- Autenticação via JWT tokens com expiração configurável
- Autorização baseada em roles com verificação em cada endpoint
- Encriptação de dados sensíveis em repouso
- HTTPS obrigatório para toda a comunicação
- Validação rigorosa de todos os inputs
- Logs de auditoria de todas as ações administrativas
A plataforma está preparada para GDPR e conformidade com políticas corporativas."

PERGUNTA 4: "Se a plataforma tem 2 pessoas a desenvolver, como é possível
entregar em 4 meses?"

RESPOSTA SUGERIDA:
"Boa observação. Conseguimos por:
- Reutilização de frameworks e bibliotecas maduras (Express, React, Flutter)
- Design simplificado e focado (sem features desnecessárias)
- Metodologia ágil com sprints de 2 semanas
- Testes automatizados desde cedo para evitar bugs custosos
- Divisão clara de responsabilidades (um focado no backend, outro no mobile/web)
- Prototipagem rápida para validar conceitos antes de implementar"

PERGUNTA 5: "Como pretendem dar manutenção e evoluir a plataforma após o
lançamento?"

RESPOSTA SUGERIDA:
"Este é um produto vivo. Planeamos:
- Recolher feedback dos utilizadores nos primeiros 3 meses
- Ciclos de melhoramento contínuos em sprints de 2 semanas
- Monitorização da plataforma em produção com alertas
- Documentação de código e arquitetura para facilitar manutenção
- Plano claro de feature roadmap para os próximos 12 meses
Exemplos de evoluções planeadas: integração com sistemas de formação,
recomendações baseadas em IA, gamificação de equipas."

PERGUNTA 6: "Qual foi o maior desafio técnico que enfrentaram?"

RESPOSTA SUGERIDA:
"O maior desafio foi sincronizar dados em tempo real entre mobile e web sem
sobrecarregar a API. Resolvemos com:
- Implementação de notificações push
- Cache inteligente no cliente
- Sincronização apenas de dados modificados (delta sync)
- WebSockets para atualizações em tempo real onde necessário
Isto garante que um utilizador no mobile vê a mesma informação que alguém na web."

PERGUNTA 7: "Como medem o sucesso da plataforma?"

RESPOSTA SUGERIDA:
"Definimos métricas claras:
- Utilização: % de colaboradores que acessa a plataforma regularmente
- Engagement: tempo médio de utilização por colaborador
- Impacto: correlação entre uso de badges e performance/satisfação
- Adopção: % de aumento de badges conquistadas ao longo do tempo
- Negócio: ROI medido em redução de turnover e aumento de produtividade
Planeamos acompanhamento mensal via dashboard que temos na plataforma."

PERGUNTA 8: "O projeto pode ser adaptado para outras empresas além da Softinsa?"

RESPOSTA SUGERIDA:
"Absolutamente. Embora tenha sido desenvolvida para a Softinsa, a arquitetura
permite:
- Customização de badges e critérios por empresa
- White-label da interface (cores, logos, branding)
- Integração com diferentes sistemas de RH
- Suporte a múltiplas estruturas organizacionais
Tecnicamente, seria uma questão de deployment em novo ambiente e customização
de configurações. O código core não precisaria alterações significativas."

NOTAS FINAIS
================================================================================

Este guião é um ROTEIRO, não um SCRIPT. Não memorizem palavra por palavra.

O objetivo é que cada apresentador compreenda profundamente o seu tópico e
consiga falar naturalmente sobre ele, como se estivessem a explicar a um colega.

A apresentação ideal é fluida, onde os ouvintes esquecem que estão a ouvir uma
apresentação preparada - parece uma conversa entre pessoas que sabem muito sobre
um assunto.

Foquem-se em:
1. Compreensão profunda do conteúdo (não memorização)
2. Transmissão de entusiasmo pelo projeto
3. Respostas claras e diretas a perguntas
4. Gestão do tempo com margem de segurança
5. Confiança na apresentação

Boa sorte! Vão estar ótimos! 🚀
