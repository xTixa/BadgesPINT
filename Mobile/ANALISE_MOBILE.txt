0. COMO USAR ESTE DOCUMENTO
================================================================================

Este documento foi escrito para quem não domina Flutter. Primeiro apresenta a
aplicação em linguagem simples, depois explica como tudo funciona tecnicamente e,
no fim, descreve os ficheiros um a um. As secções 1, 2 e 16 servem como guião de
apresentação. As restantes ajudam a responder a perguntas.

Nota de rigor: “mobile_app” é o nome técnico do pacote, mas o título mostrado pela
aplicação é “Badges Softinsa”. O projeto Flutter contém suporte para Android, iOS,
Web, Windows, Linux e macOS. A utilização principal é mobile e apenas o perfil
“consultant” (consultor) pode entrar nesta aplicação.


================================================================================
1. EXPLICAÇÃO EM 30 SEGUNDOS
================================================================================

A BadgesPINT é uma aplicação onde um consultor acompanha e demonstra as suas
competências. O consultor entra com a sua conta, consulta o catálogo de badges,
candidata-se a um badge, envia ficheiros que provam que cumpriu os requisitos e
acompanha a validação. Quando obtém o badge, recebe pontos e pode descarregar ou
partilhar o certificado. Também pode consultar o ranking, objetivos, notificações,
badges a expirar e o seu progresso profissional.

A app comunica com um servidor através de uma API REST. Guarda uma cópia dos
dados no dispositivo para continuar a mostrar informação sem rede. Quando a
ligação regressa, sincroniza novamente certas operações pendentes.

2. GUIÃO DE APRESENTAÇÃO (5 A 8 MINUTOS)
================================================================================

1. Problema
   “A organização precisa de uma forma simples de os consultores descobrirem
   competências, apresentarem provas e acompanharem a aprovação dos seus badges.”

2. Solução
   “Criámos uma aplicação Flutter para consultores. O mesmo código pode ser
   compilado para várias plataformas e a interface adapta-se ao tamanho do ecrã.”

3. Percurso principal
   “O utilizador autentica-se, explora o catálogo, abre um badge, candidata-se,
   desbloqueia os requisitos, envia uma evidência e acompanha o pedido até à
   validação do Talent Manager e da Service Line.”

4. Valor depois da aprovação
   “O badge obtido aumenta os pontos, aparece no perfil e na timeline e gera um
   certificado que pode ser descarregado, verificado por link e partilhado.”

5. Dados e arquitetura
   “A interface não fala diretamente com a API. Passa pelo controller e pelo
   repository. O repository comunica com o servidor e com os DAOs, que guardam
   a cópia local em SQLite. Esta separação facilita manutenção e testes.”

6. Funcionamento com falhas de rede
   “Os dados já sincronizados continuam disponíveis. Algumas ações pequenas,
   como marcar notificações ou criar um pedido, podem ficar numa fila. O upload
   do ficheiro de evidência necessita de Internet.”

7. Notificações
   “Firebase Cloud Messaging entrega notificações push. A app também mostra
   alertas locais para badges a expirar e objetivos com prazo próximo.”

8. Fecho
   “O resultado junta candidatura, validação, certificados, progressão e
   gamificação numa experiência única para o consultor.”

Sugestão de demonstração:
  Login → Dashboard → Explorar → abrir badge → Candidatar-me → Submeter evidência
  → Histórico/Pedidos → Perfil/Certificados → Ranking → Definições.

5. ARQUITETURA — QUEM FAZ O QUÊ
================================================================================

Fluxo simplificado:

  Pessoa toca num botão
          ↓
  Página/Widget (interface)
          ↓
  ConsultorController (estado e coordenação)
          ↓
  ConsultorRepository (regras e escolha da fonte)
       ↙                              ↘
  ApiClient → API/servidor       DAO → SQLite/cache
          ↓                           ↓
      resposta       →       estado atualizado
                                  ↓
                      notifyListeners() redesenha a UI

Camadas:
  - Apresentação: auth/, consultor/pages/ e consultor/widgets/.
  - Estado: consultor_controller.dart, baseado em ChangeNotifier.
  - Domínio/dados: consultor_models.dart e consultor_repository.dart.
  - Comunicação: api_client.dart.
  - Sincronização: sync_service.dart e mutation_queue_service.dart.
  - Persistência: database_helper.dart e data/local/*.dart.

Padrões usados: Repository, DAO, Observer/ChangeNotifier, Singleton e uma
abordagem offline-first. Não é MVC puro: é mais correto descrevê-la como uma
arquitetura em camadas com Controller + Repository + DAO.


================================================================================
6. ARRANQUE, SESSÃO E NAVEGAÇÃO
================================================================================

Arranque em main.dart:
  1. Prepara o Flutter.
  2. Em plataformas que não sejam Web, abre/cria badges_pint.db.
  3. Inicia a monitorização da conectividade.
  4. Tenta iniciar Firebase; se falhar, a app continua sem push.
  5. Arranca BadgesPintApp.

Bootstrap em app.dart:
  1. Carrega token e id do utilizador guardados.
  2. Tenta sincronizar as áreas e lê a cópia local.
  3. Inicia a fila de ações offline.
  4. Se houver sessão, carrega o controller e valida o perfil.
  5. Só aceita role == "consultant"; os restantes perfis são rejeitados.
  6. Regista o token do dispositivo no serviço de notificações.

Estados de autenticação:
  login → register ou firstLogin → authenticated.

Rotas públicas:
  /boot              ecrã temporário de carregamento
  /login             início de sessão
  /register          criação de conta de consultor
  /recover-password  pedido de recuperação
  /first-login       substituição obrigatória da password temporária

Separadores autenticados, cada um mantendo a sua própria pilha de navegação:
  /app/dashboard     Dashboard
  /app/home          Explorar catálogo
  /app/upload        Submeter evidências
  /app/history       Histórico
  /app/profile       Perfil

Rotas autenticadas auxiliares:
  /ranking, /consultants/:id, /notifications, /settings e /timeline.

O redirect do GoRouter impede páginas privadas sem sessão e impede voltar ao
login depois de autenticado. Alguns ecrãs (detalhe, pedidos e gamificação) abrem
com Navigator/MaterialPageRoute em vez de terem uma rota GoRouter própria.

Sessão:
  - O token e o id ficam em SharedPreferences.
  - O ApiClient envia Authorization: Bearer <token> quando necessário.
  - Cookies Set-Cookie da API também são guardados e reenviados.
  - No logout, remove token FCM, limpa fila, cookies, token e id.

7. FLUXOS FUNCIONAIS
================================================================================

Autenticação e conta:
  - Login com email/password e opção “lembrar credenciais”.
  - A API devolve token, utilizador e indicador de primeiro login.
  - Primeiro login obriga à escolha de nova password.
  - Registo envia nome, email, área e aceitação RGPD; a password temporária é
    enviada externamente por email pelo sistema de backend.
  - Recuperação pede ao backend o envio do email correspondente.

Candidatura a badge:
  1. O catálogo mostra nome, descrição, área, nível e pontos.
  2. O detalhe apresenta o badge; antes da candidatura, requisitos bloqueados.
  3. “Candidatar-me” cria /api/pedidos e, se o pedido estiver open, submete-o.
  4. Os requisitos ficam visíveis e surge a ação de upload.
  5. O utilizador escolhe PDF/PNG/JPG/JPEG/DOC/DOCX, no máximo 8 MB.
  6. O ficheiro é convertido para Base64, enviado e a URL devolvida é associada
     ao requisito juntamente com notas.
  7. A evidência fica pendente, aprovada ou rejeitada.
  8. O pedido acompanha validação do Talent Manager e da Service Line.

Badge obtido e certificado:
  - O badge obtido conta para progresso, pontos, perfil e gamificação.
  - O certificado PDF é pedido ao servidor e guardado pelo seletor de ficheiro;
    em Web é descarregado pelo browser; se necessário usa a pasta da aplicação.
  - Existe link público /share/badges/<id>, abertura no browser e partilha
    LinkedIn. No perfil, a partilha depende das preferências RGPD/público/LinkedIn.

Atualização:
  - O primeiro carregamento sincroniza perfil, badges, progresso, catálogo,
    pedidos, notificações, áreas, ranking, learning paths, certificados e alertas.
  - A cada 20 segundos atualiza pedidos, notificações, badges, progresso, ranking,
    learning paths, certificados e alertas.

9. MODELOS E ESTADO
================================================================================

ConsultorController guarda em memória:
  profile, badges, badgesProgress, recommendations, expiryAlerts, ranking,
  learningPaths, certificates, areas, catalogBadges, preferredAreaBadges,
  pedidosStatus, notifications, badge selecionado, requirements e evidences.
  Guarda ainda loading, estado online e número de ações pendentes.

Valores calculados:
  - badgesObtidos: badges com status “obtido”.
  - totalPoints: pontos do perfil; se forem zero, soma os badges.
  - globalProgress: percentagem obtidos/total.
  - recommendations: até quatro badges ainda não obtidos, preferindo a área.
  - rankingPosition: posição cujo consultantId coincide com o perfil.

Modelos em consultor_models.dart:
  - ConsultantUser: conta, pontos, área, localização, avatar e preferências.
  - AreaItem: id e nome da área.
  - BadgeItem: badge do utilizador, estado, pontos, área e expiração.
  - CatalogBadgeItem: badge do catálogo, descrição, nível, área e imagem.
  - BadgeProgress: requisitos totais e aprovados por badge.
  - RequirementItem: requisito, descrição, código e imagem.
  - EvidenceItem: evidência, requisito, estado e URL.
  - PedidoBadgeStatus: candidatura, workflow, datas e comentários.
  - RankingItem: pessoa e posição no ranking.
  - PublicConsultantProfile: perfil que pode ser visto por colegas.
  - LearningPathProgressItem: progresso numa rota de aprendizagem.
  - CertificateItem: certificado e URL de verificação.
  - ExpiryAlert: badge e dias até expirar.
  - UserNotificationItem: título, mensagem e estado lido.
  - RecommendationItem: recomendação construída localmente.

10. API — ENDPOINTS USADOS
================================================================================

Autenticação/conta:
  POST /api/auth/login                     autenticar
  POST /api/auth/recover-password          recuperar password
  POST /api/auth/first-login               trocar password temporária

Catálogo e badges:
  GET  /api/areas                          áreas
  GET  /badges                             catálogo
  GET  /areas/<areaId>/badges              badges por área
  GET  /badges/<badgeId>/requirements      requisitos
  GET  /api/consultor/<id>/badges          badges do consultor
  GET  /api/consultor/<id>/badges-progress progresso
  GET  /api/consultor/badges/<id>/evidencias evidências
  POST /api/consultor/evidencias/upload    upload Base64
  POST /api/consultor/requirements/<id>/evidencias associar evidência
  POST /api/consultor/badges/<id>/certificado gerar/obter PDF

Pedidos e progressão:
  GET/POST /api/pedidos                    listar/criar candidatura
  POST /api/pedidos/<id>/submeter          submeter pedido aberto
  GET  /api/consultores/ranking            ranking
  GET  /api/consultor/learning-paths/progress learning paths
  GET  /api/consultor/certificates         certificados
  GET  /api/consultor/badges-expirar       alertas até 30 dias
  GET  /api/consultor/<id>/profile         perfil público
  PUT  /api/consultor/preferences          RGPD, público, LinkedIn e objetivo

Notificações:
  GET  /api/notifications
  PUT  /api/notifications/<id>/read
  PUT  /api/notifications/mark/all-read
  POST /api/notifications/device-token
  POST /api/notifications/device-token/remove

O ApiClient considera sucesso apenas códigos 200–299, tem timeout de ligação de
12 s e de resposta de 20 s e transforma os restantes em ApiException.

11. BASE LOCAL, SINCRONIZAÇÃO E OFFLINE
================================================================================

Base nativa: badges_pint.db, versão 3. Tabelas:
  cache_records      JSON genérico por chave
  sync_metadata      última tentativa, sucesso/erro e mensagem
  pending_mutations  ações POST/PUT por enviar e número de tentativas
  users              perfil
  areas              áreas
  my_badges           badges do utilizador
  badge_progress      progresso
  catalog_badges      catálogo
  badges_by_area      catálogo por área
  applications        pedidos
  notifications       notificações
  requirements        requisitos por badge
  evidences           evidências por utilizador e badge

As listas são substituídas dentro de uma transação e conservam a ordem em
row_order. O JSON integral também é guardado, o que reduz mapeamentos SQL.

Em Web não é usado sqflite: JsonCacheDao mantém um Map em memória. Portanto, ao
recarregar a página, essa cache desaparece. A fila de mutações também é desativada
em Web.

Fila offline implementada para:
  - criar pedido;
  - associar uma evidência que já tenha URL;
  - marcar uma ou todas as notificações como lidas.

Quando a conectividade regressa, MutationQueueService reenvia em ordem. Cada ação
falhada aumenta attempts; getAll deixa de devolver itens com 5 tentativas. O item
continua na tabela, mas já não volta a ser processado automaticamente.

Limites importantes:
  - connectivity_plus confirma uma rede, não garante que a Internet/API responde.
  - O upload do ficheiro exige ligação e a interface bloqueia-o offline.
  - Outras alterações, como perfil/password/preferências, não entram na fila.
  - A sincronização captura erros e mantém dados locais antigos, permitindo leitura.

12. NOTIFICAÇÕES, TEMA E FICHEIROS
================================================================================

Push:
  NotificationService inicializa Firebase, pede permissão, obtém token FCM e
  regista-o na API. Mensagens em primeiro plano geram notificação local e atualizam
  dados. Tocar numa mensagem abre o centro de notificações. Existem canais Android
  “badges_alerts” (alta prioridade) e “badges_sla” (máxima).

Alertas locais:
  - Um aviso reúne até três badges que expiram em 0–7 dias.
  - Um lembrete de objetivo surge se o prazo estiver entre 0 e 30 dias.
  - Apesar do nome “schedule”, o código usa show(): mostra o aviso no momento em
    que a função corre, não agenda uma data futura no sistema operativo.

Tema:
  Tema claro Material 3. Azul principal #0F62FE, azul claro #00AEEF, fundo
  #F6F9FC. app_theme.dart uniformiza barras, cartões, inputs, botões e navegação.
  SessionStorage possui suporte para guardar themeMode, mas app.dart aplica sempre
  AppTheme.light(); modo escuro ainda não está ligado à interface.

Ficheiros:
  Evidências aceites: pdf, png, jpg, jpeg, doc, docx; limite 8 MB na interface.
  Certificados: PDF. Na Web usa Blob/anchor; no dispositivo usa FilePicker e depois
  a pasta de documentos como fallback.

15. PONTOS FORTES, LIMITAÇÕES E RISCOS REAIS
================================================================================

Pontos fortes:
  - Separação clara entre UI, estado, API e persistência.
  - Código comum para várias plataformas e layout responsivo.
  - Cache local, metadados de sync e fila para algumas ações offline.
  - Validação de perfil consultant, proteção de rotas e timeouts HTTP.
  - Notificações push, certificados, ranking e percurso completo de candidatura.
  - Fallback visual quando imagens de badge não funcionam.

Limitações a apresentar honestamente:
  - Só existe um teste automatizado de interface e a cobertura é muito reduzida.
  - “Lembrar password” guarda a password em SharedPreferences, sem armazenamento
    seguro como Keychain/Keystore. É um risco de segurança a corrigir.
  - Token e cookies também usam SharedPreferences; idealmente usariam secure storage.
  - O logout limpa sessão e fila, mas não chama clearUserData(); a cache SQLite do
    utilizador continua no dispositivo. Os dados devem ser limpos no logout ou
    protegidos/encriptados conforme os requisitos de privacidade.
  - app.dart aplica apenas tema claro; themeMode está preparado mas não ligado.
  - Em Web a cache é apenas memória e não sobrevive a refresh.
  - Offline é parcial e uploads precisam de rede.
  - A fila deixa de repetir após cinco falhas sem mostrar uma ação de recuperação.
  - Algumas opções das definições (email de confirmação, aprovação/rejeição,
    sugestões e assinatura de email) são guardadas localmente, mas não controlam
    serviços visíveis no código mobile atual.
  - BadgeDetail permite copiar link público mesmo antes de confirmar permissões de
    publicação; o backend deve continuar a aplicar controlo de acesso.
  - O build Android release usa assinatura de debug e contém TODO para signing.
  - google-services.json e env/dev.json contêm configuração de cliente. Chaves de
    Firebase de cliente não são passwords, mas devem ter restrições no Firebase.
  - README.md ainda é o texto genérico criado pelo Flutter.
  - Existem caracteres portugueses mal codificados em algumas strings do código
    (“EstÃ¡s”, “DefiniÃ§Ãµes”), que podem aparecer incorretamente na interface.
  - badges_page.dart e parte do suporte de tema/local_database são redundantes ou
    não usados, aumentando manutenção.

Melhorias prioritárias:
  1. Remover armazenamento da password ou migrar sessão para flutter_secure_storage.
  2. Criar testes unitários de repository/controller/sync e testes dos fluxos.
  3. Corrigir encoding UTF-8, README e textos sem acentos.
  4. Implementar assinatura release e pipeline CI.
  5. Tornar preferências realmente efetivas e mostrar falhas permanentes da fila.
  6. Persistir cache Web (IndexedDB) se Web for requisito oficial.

16. PERGUNTAS PROVÁVEIS NA APRESENTAÇÃO
================================================================================

“Porque escolheram Flutter?”
  Porque permite partilhar a maior parte do código entre Android, iOS, Web e
  desktop, mantendo uma interface consistente e desenvolvimento rápido.

“A app funciona sem Internet?”
  Parcialmente. Mostra os últimos dados sincronizados e põe algumas ações em fila.
  O upload do ficheiro e várias alterações de conta exigem ligação.

“Onde ficam os dados?”
  A fonte oficial é a API/backend. O dispositivo guarda uma cópia SQLite para
  leitura e sincronização; preferências pequenas ficam em SharedPreferences.

“Como sabe que o utilizador está autenticado?”
  Depois do login recebe um token. Esse token acompanha os pedidos protegidos e o
  router bloqueia páginas privadas quando não existe sessão válida.

“Como a interface atualiza?”
  O ConsultorController altera o estado e chama notifyListeners(). Os widgets que
  escutam o controller são reconstruídos com os novos dados.

“Como funciona a validação de um badge?”
  O consultor cria e submete um pedido, associa evidências aos requisitos e vê as
  etapas Talent Manager e Service Line. O backend é responsável pela decisão.

“As recomendações usam inteligência artificial?”
  Não no código atual. São regras simples: escolhem badges ainda não obtidos,
  preferindo a área do consultor e limitando a quatro.

“O ranking é calculado na app?”
  Não. A API devolve posições e pontos. Em modo debug existe apenas um fallback
  mock se o ranking falhar.

“As notificações vêm de onde?”
  Push vem de Firebase/servidor. Expiração e objetivo são notificações locais
  construídas com os dados que a app sincronizou.

“Porque usam SQLite e também SharedPreferences?”
  SQLite é adequado a listas e dados estruturados. SharedPreferences é mais simples
  para token e interruptores de definições; embora dados sensíveis devam migrar para
  armazenamento seguro.

“Qual é a principal dívida técnica?”
  Segurança do armazenamento local de credenciais e falta de testes automatizados.

17. RESUMO FINAL PARA DECORAR
================================================================================

  - Flutter/Dart, Material 3, aplicação principal para consultores.
  - Fluxo: explorar → candidatar → requisitos → evidências → validação → certificado.
  - GoRouter protege rotas e mantém cinco separadores autenticados.
  - ChangeNotifier gere o estado; Repository separa a UI dos dados.
  - Dio comunica com a API; SQLite e DAOs guardam a cópia local.
  - Sincroniza no arranque e faz polling de 20 em 20 segundos.
  - Fila offline cobre algumas ações; upload exige Internet.
  - Firebase gere push; notificações locais alertam expiração e objetivos.
  - Pontos, ranking, níveis e conquistas dão a componente de gamificação.
  - A arquitetura é sólida, mas segurança das credenciais, testes, encoding e
    configuração de release são os próximos trabalhos prioritários.
